Docker - continuous deployment and virtualisation

  • spans virtualization and deployment (config. management applications)
  • very new, fast success; aids communication success in organisations' teams
  • devs must understand about the complexity of the environment they are shipping into (must they?) and conversely prod. ops need to understand about the internals of what they are shipping
  • reduces need for devs to ask for a release 1.1 of a library removes a bottleneck source.
  • instead devs can directly update the resource set they need
  • Docker discards entire environment of running application to make way for a new one
    • makes less chance for cruft causing problems
    • makes things ultra portable
    • thus more scalable, more reliable
  • builds myriad dependencies in one single, repeatable format. Deploy app with every single file that is needed to run it. Means app is running in the environment expect it to run in.
  • doesn't need to replace older systems : can bind to these easily and straightforwardly, for different needs.

Containers

  • old linux tech, but renewed in powerful way by the choices Docker has made.
  • diff. to virtualization - where different OS run on the host OS - Docker containers use the same kernel as the host(i.e. linux). Share resources this way so better.
  • makes container images, encapsulating all dependencies.
  • diff to Vagrant - only 1 OS, (V is more for devs)

Vagrant and Virtualization

Build, Manage, Distribute Test & Dev. environments

  • Combines with advent of the Cloud (virtualisation is foundation for AWS, Azure, and eg Linode slices etc) and brings it to Desktops.
  • Build and configure a virtual machine for any need.
  • Vagrant 1.0 released 2012, so relatively mature now, v2.0.x and each release is pretty stable
  • encourages automation of dev. environments
  • reproducible environments enables team working - just share the image
  • devs can still develop with their own tools and editors etc, but in a consistent rapid environment.
  • uses VirtualBox but can work with EC2, VMWare, Parallels etc

  • convergence of the dev. to the prod. environment parameters with the automation - valuable change

    vagrant up

Something amiss? Simply 

vagrant destroy

and start over.
